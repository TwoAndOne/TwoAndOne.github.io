注解的本质就是一个继承了Annotation接口的接口

元注解

注解注解的注解（中国语言博大精深，充分利用词语的词性）

@Target：指明作用目标

ElementType
ElementType.TYPE：允许被修饰的注解作用在类、接口和枚举上
ElementType.FIELD：允许作用在属性字段上
ElementType.METHOD：允许作用在方法上
ElementType.PARAMETER：允许作用在方法参数上
ElementType.CONSTRUCTOR：允许作用在构造器上
ElementType.LOCAL_VARIABLE：允许作用在本地局部变量上
ElementType.ANNOTATION_TYPE：允许作用在注解上
ElementType.PACKAGE：允许作用在包上

@Retention：指明当前注解的生命周期

  • RetentionPolicy.SOURCE：当前注解编译期可见，不会写入 class 文件
  • RetentionPolicy.CLASS：类加载阶段丢弃，会写入 class 文件
  • RetentionPolicy.RUNTIME：永久保存，可以反射获取

@Documented 注解修饰的注解，当我们执行 JavaDoc 文档打包时会被保存进 doc 文档，反之将在打包时丢弃。

@Inherited 注解修饰的注解是具有可继承性的，也就说我们的注解修饰了一个类，而该类的子类将自动继承父类的该注解。

1 开篇陈词

一切为了提高性能！

为了不浪费等待IO的CPU资源，所以我们把任务拆分交替执行分时操作系统，这就是并发。

随着技术的发展，出现了多核CPU，我们就可以实现并行计算。

为了充分利用多核CPU，进一步提升效率，我们需要对任务进行组织编排，进行分工。对线程进行分工了之后，这就导致一个问题，有一些任务是有依赖关系的，要想实现这些依赖关系，就得让线程之间进行通信，也就是同步。

当然为了并发和通信，也引入了新的问题，那就是多线程并发操作临界资源的问题。

对于临界资源，我们的解决方案是锁、信号量等，这就是互斥。

一切为了提高性能！

那么问题有哪些呢？

  • 缓存——可见性问题
  • 线程切换——原子性问题
  • 编译优化——有序性问题

那么Java是如何解决这些问题的呢？

如果说缓存和编译优化会带来可见性和有序性的问题，是不是直接禁用缓存和编译优化就行了。但是如果直接禁用，程序的性能就堪忧了，所以我们需要按需禁用。

Java内存模型规范了JVM如何提供按需禁用缓存和编译优化的方法：

  • volatile
  • synchronize
  • final
  • 六个Happens-Before规则

解决原子性问题，是要保证中间状态对外不见

用细粒度锁来锁定多个资源时，要注意死锁的问题

死锁的四个条件

  • 互斥

  • 占有且等待

  • 不可抢占

  • 循环等待

wait与sleep区别在于：

  1. wait会释放所有锁而sleep不会释放锁资源.
  2. wait只能在同步方法和同步块中使用，而sleep任何地方都可以.
  3. wait无需捕捉异常，而sleep需要.

两者相同点：都会让渡CPU执行时间，等待再次调度！

7 JUC

java 并发包通过Lock和Condition两个接口来实现管程，其中Lock解决互斥问题，Condition解决同步问题。

已经有了synchronized，为什么还要设计一个Lock接口? 因为synchronized会让线程直接进入blocked状态，如果陷入死锁，那么就无法解除了。

锁的最佳实践

  • 永远只在更新对象的成员变量时加锁
  • 永远只在访问可变的成员变量时加锁
  • 永不在调用其他对象的方法时加锁
  • ———-《Java并发编程：设计原则与模式》Doug Lea 减少锁的持有时间 减小锁的粒度

7 Concurrent包

CompletionService

作用

与CompletableFuture的区别在于这里可以批量执行异步任务。执行之后还可以拿到先完成的任务结果，这里是用一个阻塞队列实现的。

使用

什么时候使用

当需要批量提交异步任务的时候。CompletionService将线程池Executor和阻塞队列BlockQueue的功能融合在了一起，能够让批量异步任务的管理更简单。 而且能够让异步任务的执行结果有序化，先执行的先进入阻塞队列，利用这个特性可以避免无谓的等待。

对于简单的并行任务，可以通过“线程池+Future”的方案解决；如果任务之间有聚合关系，可以通过CompletableFuture解决；而批量的并行任务，则可以通过CompletionService解决。

全排列的实质是遍历多叉树,假如数组是[1,2,3]

所以我们的思路就是

  • 终止条件
    • 遍历到最后一个，也就是数组长度等于3的时候
  • 使用循环语句把每一个分支遍历到
  • 在循环语句内部使用递归调用本函数接着遍历子树
  • 同一层的结点遍历结束后需要回溯，举个例子，当遍历到[1,1,1]时下一个应该遍历[1,1,2]，此时在添加了[1,1,1]之后需要把最后一个1删除以增加空位。

当然这只是第一步，得到全排列还需要剪枝，因为[1,1,1]这种排列并不是我们需要的，不过我们可以先实现这一种

问题来了，我们怎么剪枝呢？

简单的方法就是，我们在tmp.add(nums[i])之前进行判断tmp中是否已经有这个数了，如果有了就continue。

还有一种方法是用一个标记数组进行记录被使用过的数字，这种也是回溯算法常用的方式

好的，全排列是大功告成啦！

但是子集还没搞定

先从78. 子集开始

从示例中可以看出，子集会把tmp数组每一次变化加入list中，而全排列则需要在tmp数组长度为3时才加入

总结一下：

  • 终止条件
    • tmp数组长度等于nums.length
  • tmp数组的每一次变化都需要加入进返回数组list
  • 循环语句中，子集的剪枝可以从示例中看出，子集不需要回头

子集Ⅱ与子集的区别就是给定的数组里有重复的数字

加入给定数组[1,2,2]，不能出现，[1,2],[1,2]的情况，也就是挑选了下标为0，1和下标为0，2的的情况是一样的，不能出现这种情况

容易想到的方法是，先对数组进行排序，如果nums[i]与nums[i-1]相等就跳过下标为i的情况

考虑到边界条件，i必须大于0，但是只考虑这两种[1,2,2]的情况就会被排除，所以必须让for循环至少执行一次，也就是i大于index并且nums[i]与nums[i-1]相等时才不考虑i这种情况，当i等于index会保证至少执行一次。

分析

new关键字用于实例化一个对象，而实例化一个对象首先要确保一个类已经被加载进JVM中

类加载的过程分为

  1. 加载：生产Class对象，存在方法区

  2. 连接

    1. 验证：文件格式、元数据、字节码、符号引用验证
    2. 准备：为类变量分配内存，也就是被static关键字修饰的变量
    3. 解析：将常量池的符号引用替换成直接引用，也就是得到类或字段、方法在内存中的指针或者偏移量

  3. 初始化：如果父类未初始化，会先触发父类的初始化，初始化会按照代码顺序依次执行静态变量和静态块，如下图，在静态块中引用未初始化的静态变量会编译失败。

总结

所以运行结果是输出A类的静态块接着输出B类的静态块，然后因为类已经加载进来了，所以第二次new的时候就不需要执行被static修饰的部分了

实例化对象时，通过构造方法实现，如果有父类需要先实例化父类再实例化子类。
