对原型模式的思考

Posted on 2022-08-09 Edited on 2023-08-22 In tech

今天学了设计模式-原型模式，解释如下：

原型模式是创建型模式的一种，其特点在于通过「复制」一个已经存在的实例来返回新的实例,而不是新建实例。 被复制的实例就是我们所称的「原型」，这个原型是可定制的。 原型模式多用于创建复杂的或者耗时的实例，因为这种情况下，复制一个已经存在的实例使程序运行更高效；或者创建值相等，只是命名不一样的同类数据。

———Wiki

geekbang: 如果对象的创建成本比较大，而同一个类的不同对象之间差别不大（大部分字段都相同），在这种情况下，我们可以利用对已有对象（原型）进行复制（或者叫拷贝）的方式来创建新对象，以达到节省创建时间的目的。这种基于原型来创建对象的方式就叫作原型设计模式（Prototype Design Pattern），简称原型模式。

———Geekbang

在脑海中能想到好几种与之相关的实现，但我不是很理解为什么这会成为一种设计模式。

fork

从词义上理解，fork是叉子的意思，很容易想到一个分支有很多分叉的样子。

在GitHub中，fork表示拷贝到自己仓库

在Linux中也有类似的意义：运行着的程序分成2个完全一样的进程，每个进程都启动一个从代码同一位置开始执行的线程，这两个进程的线程继续执行，就像是两个用户同时启动了该程序的两个副本。

输出结果：

  • fork作为一个函数被调用。这个函数会有两次返回，将子进程的PID返回给父进程，0返回给子进程。(如果小于0，则说明创建子进程失败)。
  • 再次说明：当前进程调用fork()，会创建一个跟当前进程完全相同的子进程(除了pid)，所以子进程同样是会执行fork()之后的代码。

所以说：

  • 父进程在执行if代码块的时候，fpid变量的值是子进程的pid
  • 子进程在执行if代码块的时候，fpid变量的值是0

简单解释一下为什么fpid会有不同的返回值

我们可以把创建进程的过程想象成一棵树🌲，root创建一个子节点，root就可以指向这个子节点，那么这个fpid就是子进程的进程id，因为子进程没有子进程，所以它的fpid是0，相当于叶子节点。

子进程可以根据getppid取得父进程的id，但是父进程可以有许多子进程，所以就无法获得各个子进程的进程id。

父进程中调用fork之前打开所有的描述字在fork返回之后由子进程分享。我们将看到网络服务器便利用了这个特性：父进程调用accept之后调用fork。所接受的已连接的套接口随后就在父进程与子进程之间分享。通常情况下，子进程接着读和写这个套接口，父进程则关闭这个已连接套接口。

fork的用法： 1 一个进程创建一个自身的拷贝，这样每个拷贝都可以在另一个拷贝执行其他任务的同时处理各自的某个操作。这是网络服务器的典型用法。 2 一个进程想要执行另一个程序。既然创建新进程的唯一方法为调用fork，该进程于是首先调用fork创建一个自身的拷贝，然后其中一个拷贝（通常为子进程）调用exec把自身替换成新的程序。这是诸如shell之类程序的典型用法。

对于第二种用法，创建子进程时复制过去的数据没有任何作用，因为exec会清空原来的数据。

所以就有了Copy On Write！

fork创建出的子进程，与父进程共享内存空间。也就是说，如果子进程不对内存空间进行写入操作的话，内存空间中的数据并不会复制给子进程，这样创建子进程的速度就很快了！(不用复制，直接引用父进程的物理空间)。相当于先浅拷贝，需要写的时候再深拷贝。

并且如果在fork函数返回之后，子进程第一时间exec一个新的可执行映像，那么也不会浪费时间和内存空间了。

fork()之后，kernel把父进程中所有的内存页的权限都设为read-only，然后子进程的地址空间指向父进程。当父子进程都只读内存时，相安无事。当其中某个进程写内存时，CPU硬件检测到内存页是read-only的，于是触发页异常中断（page-fault），陷入kernel的一个中断例程。中断例程中，kernel就会把触发的异常的页复制一份，于是父子进程各自持有独立的一份。

COW在文件系统中也有应用，当我要修改数据块A的内容的时候，我先把A读出来，写到B块里，如果写的过程掉电了，原来A的内容还在，如果是还写到原来的位置上，那么写入的数据究竟写了多少就不确定了，会不会破坏原来的数据也不好说。

ElementType
ElementType.TYPE：允许被修饰的注解作用在类、接口和枚举上
ElementType.FIELD：允许作用在属性字段上
ElementType.METHOD：允许作用在方法上
ElementType.PARAMETER：允许作用在方法参数上
ElementType.CONSTRUCTOR：允许作用在构造器上

ElementType.PACKAGE：允许作用在包上

@Retention：指明当前注解的生命周期

  • RetentionPolicy.SOURCE：当前注解编译期可见，不会写入 class 文件
  • RetentionPolicy.CLASS：类加载阶段丢弃，会写入 class 文件
  • RetentionPolicy.RUNTIME：永久保存，可以反射获取

@Documented 注解修饰的注解，当我们执行 JavaDoc 文档打包时会被保存进 doc 文档，反之将在打包时丢弃。

@Inherited 注解修饰的注解是具有可继承性的，也就说我们的注解修饰了一个类，而该类的子类将自动继承父类的该注解。

1 开篇陈词

一切为了提高性能！

为了不浪费等待IO的CPU资源，所以我们把任务拆分交替执行分时操作系统，这就是并发。

随着技术的发展，出现了多核CPU，我们就可以实现并行计算。

为了充分利用多核CPU，进一步提升效率，我们需要对任务进行组织编排，进行分工。对线程进行分工了之后，这就导致一个问题，有一些任务是有依赖关系的，要想实现这些依赖关系，就得让线程之间进行通信，也就是同步。

当然为了并发和通信，也引入了新的问题，那就是多线程并发操作临界资源的问题。

对于临界资源，我们的解决方案是锁、信号量等，这就是互斥。

一切为了提高性能！

那么问题有哪些呢？

  • 缓存——可见性问题
  • 线程切换——原子性问题
  • 编译优化——有序性问题

那么Java是如何解决这些问题的呢？

如果说缓存和编译优化会带来可见性和有序性的问题，是不是直接禁用缓存和编译优化就行了。但是如果直接禁用，程序的性能就堪忧了，所以我们需要按需禁用。

Java内存模型规范了JVM如何提供按需禁用缓存和编译优化的方法：

  • volatile
  • synchronize
  • final
  • 六个Happens-Before规则

wait与sleep区别在于：

  1. wait会释放所有锁而sleep不会释放锁资源.

java 并发包通过Lock和Condition两个接口来实现管程，其中Lock解决互斥问题，Condition解决同步问题。

已经有了synchronized，为什么还要设计一个Lock接口? 因为synchronized会让线程直接进入blocked状态，如果陷入死锁，那么就无法解除了。

锁的最佳实践

  • 永远只在更新对象的成员变量时加锁
  • 永远只在访问可变的成员变量时加锁
  • 永不在调用其他对象的方法时加锁

7 Concurrent包

CompletionService

作用

与CompletableFuture的区别在于这里可以批量执行异步任务。执行之后还可以拿到先完成的任务结果，这里是用一个阻塞队列实现的。

使用

什么时候使用

当需要批量提交异步任务的时候。CompletionService将线程池Executor和阻塞队列BlockQueue的功能融合在了一起，能够让批量异步任务的管理更简单。 而且能够让异步任务的执行结果有序化，先执行的先进入阻塞队列，利用这个特性可以避免无谓的等待。

对于简单的并行任务，可以通过“线程池+Future”的方案解决；如果任务之间有聚合关系，可以通过CompletableFuture解决；而批量的并行任务，则可以通过CompletionService解决。

上半年主要做的是协助用户引流组做相关各种钩子和靠近的需求，下半年主要做主站各模块的优化、审核、打折的需求。 业务需求完成度100%，除产品变更需求外，没有发生延期，需求上线bug为0，无线上重大事故。下半年粗略估计排查了15个线上问题（只看了写了排期的，有些忘记写了），粗略估计处理了18次客诉，重构了引流钩子的代码和打折push与短信发送的代码（因为这两种需求做的最多）。能够从需求中洞悉产品的核心诉求，找到方便自己也方便产品的解决方案。

个人成长

  1. 完善了自己的git提交规范 每次在排查问题时，找到当时的git commit很多都是无意义的fix，至少把当时需求的id留下也好呀。 目前使用的格式是： 增加：tapdId#feat:增加了xx功能 修复bug：tapdId#fix：修复了某个bug 文案或格式：tapdId#style：xx文案改动
  2. 优化打折短信发送逻辑 之前每次测试打折短信时，测试反馈收到短信的概率并不是百分百。这里取短信配置进行发送难点在于，目前不知道短信配置是否已经在队列中，其中等待的时间是一个比较难估计的时长。太长也许影响线上发送短信的时间，太短就会造成测试人员测试的时候拿不到配置就结束流程了。 可以把拿不到数据的情况想象成出现冲突，那么这种情况可以套用计算机网络中常用的算法“二进制指数退避算法”，按照这种思想改造优化后，测试再也没反馈过短信收不到了。 3、对MongoDB、ELK、Canal等技术可以知其然，看了fastjson的源码

现有问题

  1. 不够自信

    之前写了一个记录操作记录的切面，已经测试通过，但是自我感觉不太实用且无可用之地就一直没有同步到master。

  2. 不懂拒绝 产品需要帮忙排查逻辑的时候，总是满口答应，然后会耽误自己完成需求的时间。

  3. 思维假性近视 在学习的时候，倾向于快速得出结论“这个我早知道了”，重新想起一星半点的关键词就停止思考，将真正的知识拒之门外，甚至觉得自己已经融会贯通。

未来展望

其实学会写代码并不难，但学会写好代码却很难。 从易阅读上来说代码要有准确的命名和清晰的注释、从易使用上来说代码要具备设计模式的包装让对外的服务调用更简单、从易扩展上来说代码要做好业务和功能的实现分层。 在易阅读、易使用、易扩展以及更多编码规范的约束下，还需要在开发完成上线后的交付结果上满足:高可用、高性能、高并发，与此同时还会接到现有项目中层出不穷来自产品新增的需求。 下一年我会保持对自己的约束，不仅写代码，还写好代码。并且针对自己现有问题一一治疗。

渐渐地我就有了一种恐慌，时间过去了，生命在流逝，可我仍呆在原地，没有什么东西可以证明我随着时间一起前行。我每天吃了，喝了，睡了，总之，活下来了，可这活下来也就是活下来而已，没有获得超出活下来的意义。我一旦问自己一辈子就这样下去吗？就心里发疼，不敢再往深里想。闲着的时候，那种无聊的感觉追逐着我，紧紧地追逐着我，使我不敢面对自己。

——《沧浪之水》

为学日益，为道日损，损之又损，以至于无为。

——《老子》

“为学日益”，什么叫学问？学问是靠知识、读书、经验，一点一滴慢慢累积起来的。今天懂一点，明天再懂一点，后天又懂一点，多一分努力就多一分的收获，这就是做学问。人为的学问是有为法，是有为之道，要慢慢累积增加起来，不是一步登天。

“为道日损”，学道与做学问相反，是要丢掉，“日损”就是一天丢一点，明天再丢一点，什么都要放下丢掉。修道的人，经常笑自己，一方面有欲望学道，一方面又不肯放弃读书，爱读书就是最大的欲望。

“损之又损，以至于无为”，一切都空，空到了最后连空也把它空掉，空到一无所有；然后无所不有，一切皆知，一切皆有，就是这个简单的道理。文字很简单，意义也很简单，一说就明白了。问题是，做起来很难！如何能够把自己损之又损，放弃了又放弃，放到了一无所有之处，才能到达无所不知无所不有的境界！

老子在这里清楚地告诉我们，人生在世能够学问成就，或修道成就，就要有两种能力：“提得起”是做学问要“为学日益”；“放得下”是修道要“为道日损”，一切放下。但是普通一个人，能够具备这两种能力，两种智慧，两种勇气，所谓智勇双全，就太难了。普通的人，叫他做学问，才用功读了一个礼拜的书，便觉得很累，就停下来去玩了，为学不能日益。去修道做工夫的话，放不下，刚打坐几天，又觉得一天到晚坐着，淡而无味，浪费时间，也要跑出去玩玩，所以“为道日损”也做不到。因此，一般人多半都在为学未益、为道未损的情况下，提也提不起，放也放不下，就那样过了一生。这就是我们读了《老子》以后，自己应该反省的地方。
