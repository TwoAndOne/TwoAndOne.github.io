setAccessible(true)之我见

Posted on 2022-07-26 In tech

最近写的代码用了Field，介于经常在看源码的时候看到

在今天以前我都以为是用这个方法来突破权限访问控制，还生出：既然可以用反射来突破权限访问控制，为什么要设计这个功能的疑问。

实际上这个方法只是安全管理器的开关，为true的时候就关闭检查。

我们来跟踪一下代码，以Field为例：

可以看到 flag 被传入到了AccessibleObject当前对象的overide属性

再来看 Field ，Field类继承了AccessibleObject类，也就继承了成员变量override。

在get方法中，就判断了override的值，如果为true 就不会执行一系列check。

实际上修饰符为 public的属性或者方法的 Accessible仍然是false的，我们之所以有所误解，在于修饰符如果为public的确可以不检查，Reflection.quickCheckMemberAccess(clazz, modifiers)方法先对public进行了检查，如果是public就不进行后续的检查了。

看到这里，很明显，setAccessible(true)的作用是关闭安全检查

由于JDK的安全检查耗时较多，所以通过setAccessible(true)的方式关闭安全检查也可以提升反射速度

对文首的疑问，我是这样理解的：

Java的权限访问控制，不是为了表明“这里你一定不能修改！”，而是为了给不懂这个类具体实现的使用者，一个安全使用的方式。

而且代码是死的，而人是活的，死代码防不了活人。

注解的本质就是一个继承了Annotation接口的接口

元注解

注解注解的注解（中国语言博大精深，充分利用词语的词性）

@Target：指明作用目标

ElementType
ElementType.TYPE：允许被修饰的注解作用在类、接口和枚举上
ElementType.FIELD：允许作用在属性字段上
ElementType.METHOD：允许作用在方法上
ElementType.PARAMETER：允许作用在方法参数上
ElementType.CONSTRUCTOR：允许作用在构造器上
ElementType.LOCAL_VARIABLE：允许作用在本地局部变量上
ElementType.ANNOTATION_TYPE：允许作用在注解上
ElementType.PACKAGE：允许作用在包上

@Retention：指明当前注解的生命周期

  • RetentionPolicy.SOURCE：当前注解编译期可见，不会写入 class 文件
  • RetentionPolicy.CLASS：类加载阶段丢弃，会写入 class 文件
  • RetentionPolicy.RUNTIME：永久保存，可以反射获取

@Documented 注解修饰的注解，当我们执行 JavaDoc 文档打包时会被保存进 doc 文档，反之将在打包时丢弃。

@Inherited 注解修饰的注解是具有可继承性的，也就说我们的注解修饰了一个类，而该类的子类将自动继承父类的该注解。

在某个目录（我是在～/exp）下创建一个文件：login-it_zhenaioa_com.exp vim login-it_zhenaioa_com.exp 文件内容如下：

  1. [#!/usr/bin/expect］ 这一行告诉操作系统脚本里的代码使用那一个shell来执行。这里的expect其实和linux下的bash、windows下的cmd是一类东西。 注意：这一行需要在脚本的第一行。
  2. ［set timeout 30］ 基本上认识英文的都知道这是设置超时时间的，现在你只要记住他的计时单位是：秒
  3. ［spawn ssh -l username 192.168.1.1］ spawn是进入expect环境后才可以执行的expect内部命令，如果没有装expect或者直接在默认的SHELL下执行是找不到spawn命令的。所以不要用 “which spawn“之类的命令去找spawn命令。好比windows里的dir就是一个内部命令，这个命令由shell自带，你无法找到一个dir.com 或 dir.exe 的可执行文件。 它主要的功能是给ssh运行进程加个壳，用来传递交互指令。
  4. ［expect "password:"］ 这里的expect也是expect的一个内部命令，有点晕吧，expect的shell命令和内部命令是一样的，但不是一个功能，习惯就好了。这个命令的意思是判断上次输出结果里是否包含“password:”的字符串，如果有则立即返回，否则就等待一段时间后返回，这里等待时长就是前面设置的30秒
  5. ［send "ispass\r"］ 这里就是执行交互动作，与手工输入密码的动作等效。 温馨提示： 命令字符串结尾别忘记加上 “，如果出现异常等待的状态可以核查一下。
  6. ［interact］ 执行完成后保持交互状态，把控制权交给控制台，这个时候就可以手工操作了。如果没有这一句登录完成后会退出，而不是留在远程终端上。如果你只是登录过去执行一段命令就退出，可改为［expect eof］

Muti Thread

Posted on 2022-02-24 Edited on 2022-07-18 In tech

1 开篇陈词

一切为了提高性能！

为了不浪费等待IO的CPU资源，所以我们把任务拆分交替执行分时操作系统，这就是并发。

随着技术的发展，出现了多核CPU，我们就可以实现并行计算。

为了充分利用多核CPU，进一步提升效率，我们需要对任务进行组织编排，进行分工。对线程进行分工了之后，这就导致一个问题，有一些任务是有依赖关系的，要想实现这些依赖关系，就得让线程之间进行通信，也就是同步。

当然为了并发和通信，也引入了新的问题，那就是多线程并发操作临界资源的问题。

对于临界资源，我们的解决方案是锁、信号量等，这就是互斥。

一切为了提高性能！

那么问题有哪些呢？

  • 缓存——可见性问题
  • 线程切换——原子性问题
  • 编译优化——有序性问题

那么Java是如何解决这些问题的呢？

如果说缓存和编译优化会带来可见性和有序性的问题，是不是直接禁用缓存和编译优化就行了。但是如果直接禁用，程序的性能就堪忧了，所以我们需要按需禁用。

Java内存模型规范了JVM如何提供按需禁用缓存和编译优化的方法：

  • volatile
  • synchronize
  • final
  • 六个Happens-Before规则

2 死锁

@test.MutiThread0

加锁本质就是在锁对象的对象头中写入当前线程id

解决原子性问题，是要保证中间状态对外不见

用细粒度锁来锁定多个资源时，要注意死锁的问题

死锁的四个条件

  • 互斥

  • 占有且等待

  • 不可抢占

  • 循环等待

wait与sleep区别在于：

  1. wait会释放所有锁而sleep不会释放锁资源.
  2. wait只能在同步方法和同步块中使用，而sleep任何地方都可以.
  3. wait无需捕捉异常，而sleep需要.

两者相同点：都会让渡CPU执行时间，等待再次调度！

安全问题

数据竞争： 多个线程同时访问一个数据，并且至少有一个线程会写这个数据。 竞态条件： 程序的执行结果依赖程序执行的顺序。 也可以按照以下的方式理解竞态条件： 程序的执行依赖于某个状态变量，在判断满足条件的时候执行，但是在执行时其他变量同时修改了状态变量。 if (状态变量 满足 执行条件) { 执行操作 }

7 JUC

java 并发包通过Lock和Condition两个接口来实现管程，其中Lock解决互斥问题，Condition解决同步问题。

已经有了synchronized，为什么还要设计一个Lock接口? 因为synchronized会让线程直接进入blocked状态，如果陷入死锁，那么就无法解除了。

锁的最佳实践

  • 永远只在更新对象的成员变量时加锁
  • 永远只在访问可变的成员变量时加锁
  • 永不在调用其他对象的方法时加锁
  • ———-《Java并发编程：设计原则与模式》Doug Lea 减少锁的持有时间 减小锁的粒度

7 Concurrent包

CompletionService

作用

与CompletableFuture的区别在于这里可以批量执行异步任务。执行之后还可以拿到先完成的任务结果，这里是用一个阻塞队列实现的。

使用

什么时候使用

当需要批量提交异步任务的时候。CompletionService将线程池Executor和阻塞队列BlockQueue的功能融合在了一起，能够让批量异步任务的管理更简单。 而且能够让异步任务的执行结果有序化，先执行的先进入阻塞队列，利用这个特性可以避免无谓的等待。

对于简单的并行任务，可以通过“线程池+Future”的方案解决；如果任务之间有聚合关系，可以通过CompletableFuture解决；而批量的并行任务，则可以通过CompletionService解决。

1 进程与线程的基本概念

1.1 进程产生的背景

最开始计算机是处理完一个任务，再处理下一个任务，有些时候一些任务io时间很长，傻乎乎的等会很浪费CPU资源。

有没有一种方式可以让一个任务在等待IO时，把CPU让给另一个任务呢？

所以进程出现了。给每个进程分配一块内存，分配一个时间段，如果进程在时间片结束前阻塞或结束，则CPU立即切换，不用等待时间片用完。进程暂停时，会保存当前进程高状态（进程标识、进程使用的资源）

由于计算机不停发展，用户对同时使用多个任务的需要增加，比如一个浏览器进程，要开多个任务一个听音乐，一个看小说。然后还出现了多核处理器，可以真正的实现并行操作。

线程允许将任务分解为多个可并发执行的子任务，并利用多核处理器等硬件资源，实现更高效的并发性和性能。

因为线程更轻量级，而且能够共享进程的资源，适合在需要高并发和快速响应的情况下使用。

进程时操作系统资源分配的基本单位，线程时操作系统进行调度的基本单位

1.2 上下文切换

上下文是指某一时间点CPU寄存器和PC的内容

一个进程切换到另一个进程

上下文切换是计算密集型，所以线程不是越多越好。

1.3 什么是线程安全

当多个线程同时访问一个对象时，如果不用考虑这些线程在运行时环境下的调度和交替执行，也不需要进行额外的同步，或者在调用方进行任何其他的协调操作，调用这个对象的行为都可以获得正确的结果，那就称这个对象是线程安全的。

如果根本不存在多线程，又或者一段代码根本不会与其他线程共享数据，那么从线程安全的角度看，程序是串行的还是多线程执行的毫无区别。

按照“安全程度”来区分共享数据有五类：

  • 不可变：不可变对象一定是线程安全的，比如被final修饰的变量
  • 绝对线程安全
  • 相对线程安全
  • 线程兼容
  • 线程对立

1.4 线程安全的实现方式

互斥同步

同步是指在多个线程并发访问共享数据时，保证共享数据在同一个时刻只被一条线程使用

互斥是实现同步的一种手段，临界区（Critical Section）、互斥量（Mutex）、信号量（Semaphore）都是常见的互斥实现方式。

synchronized关键字，是一种块结构的同步语法。经过编译之后，会在同步块的前后分别形成monitorenter 和monitorexit这两个字节码命令。

这两个字节码指令都需要一个reference类型的参数来指明要锁定和解锁的对象。

  • 可重入
  • 重量级需要内核切换

重入锁是Lock接口最常见的一种实现

我们使用Runnable和Thread来创建一个新的线程。但是它们有一个弊端，就是run方法是没有返回值的。而有时候我们希望开启一个线程去执行一个任务，并且这个任务执行完成后有一个返回值。

JDK提供了Callable接口与Future接口为我们解决这个问题，这也是所谓的“异步”模型。

ExecutorService可以使用submit方法来让一个Callable接口执行。它会返回一个Future，我们后续的程序可以通过这个Future的get方法得到结果。

5 线程间的通信

5.1 锁与同步

在Java中锁是基于对象的，我们称它为对象锁。一个锁同一时间只能被一个线程持有。

同步是为了让线程之间按照一定的顺序执行

5.2 等待/通知机制

基于锁的方式，线程需要不断去尝试获得锁，如果失败了，再继续尝试，可能会耗费CPU资源。

等待/通知机制是另一种方式

Object类的wait()方法和notify(), notifyAll()方法来实现的。

notify()方法会随机叫醒一个正在等待的线程，而notifyAll()会叫醒所有正在等待的线程。

notify只是叫别人不要等了，但并没有释放锁，也就是让某个线程从waiting队列中唤醒，然后进入runnable队列，由于它想获得锁，所以又进入blocked队列。如果这时候持有锁的线程调用wait方法，它进入waiting队列，blocked队列的线程就可以获得锁进入runnable队列了。

需要注意的是等待/通知机制使用的是使用同一个对象锁，如果你两个线程使用的是不同的对象锁，那它们之间是不能用等待/通知机制通信的。

5.3 信号量

使用volatile关键词实现信号量通信

等待/通知机制适合两个线程之间的合作，信号量机制适合多个线程之间的合作

5.5.3 ThreadLocal类

ThreadLocal是一个本地线程副本变量工具类，内部是一个弱引用的Map来维护。

虽然两个线程使用的同一个ThreadLocal实例（通过构造方法传入），但是它们各自可以存取自己当前线程的一个值。

ThreadLocal 的常见应用场景：

  1. 线程上下文传递：ThreadLocal 可以在多个方法或组件之间传递线程相关的上下文信息，而无需显式地传递参数。每个线程都可以在 ThreadLocal 中设置和获取自己的上下文信息，从而实现线程级别的数据共享。
  2. 线程安全的对象存储：在需要存储线程安全的对象时，可以将对象存储在 ThreadLocal 中。由于每个线程都拥有独立的 ThreadLocal 实例，因此可以安全地将对象存储在其中，而不必担心线程之间的竞争条件。
  3. 数据库连接管理：在多线程的数据库访问场景中，可以使用 ThreadLocal 存储数据库连接对象。每个线程都可以从 ThreadLocal 中获取自己的连接对象，避免了线程之间的竞争和资源争用。
  4. 用户会话管理：在 Web 应用程序中，可以使用 ThreadLocal 存储用户会话信息。每个线程都可以通过 ThreadLocal 获取自己的会话对象，而不必共享会话对象或进行同步操作。
  5. 日志跟踪：ThreadLocal 可以用于在多线程环境中跟踪日志信息。每个线程可以在 ThreadLocal 中记录自己的日志上下文，从而实现线程级别的日志记录和跟踪。

需要注意的是，使用 ThreadLocal 需要谨慎考虑内存泄漏和资源释放的问题。由于 ThreadLocal 的生命周期与线程绑定，如果没有及时清理 ThreadLocal 中的数据，可能会导致内存泄漏和资源浪费。因此，在使用 ThreadLocal 时，需要注意及时清理和释放线程局部变量。

5.5.4 InheritableThreadLocal

InheritableThreadLocal类与ThreadLocal类稍有不同，Inheritable是继承的意思。它不仅仅是当前线程可以存取副本值，而且它的子线程也可以存取这个副本值。

6.2 Java内存模型的抽象结构

6.2.1 运行时内存划分

栈都是线程私有的数据，二堆是线程共享的数据，由此产生内存可见性的问题

6.2.2 既然堆是共享的，为什么堆存在内存不可见问题？

现代计算机为了高效，几乎在每个位置都设置了缓存，比如CPU有多级缓存、应用程序有缓存、数据库有缓存、分布式缓存、CDN缓存。

JMM、Java内存模型也是一个缓存。线程之间的共享变量存在主内存，每个线程都有私有的本地内存，存储了线程读写共享变量的副本。

从图中可以看出：

  1. 所有的共享变量都存在主内存中。
  2. 每个线程都保存了一份该线程使用到的共享变量的副本。
  3. 如果线程A与线程B之间要通信的话，必须经历下面2个步骤：
    1. 线程A将本地内存A中更新过的共享变量刷新到主内存中去。
    2. 线程B到主内存中去读取线程A之前已经更新过的共享变量。

根据JMM的规定，线程对共享变量的所有操作都必须在自己的本地内存中进行，不能直接从主内存中读取。

所以线程B并不是直接去主内存中读取共享变量的值，而是先在本地内存B中找到这个共享变量，发现这个共享变量已经被更新了，然后本地内存B去主内存中读取这个共享变量的新值，并拷贝到本地内存B中，最后线程B再读取本地内存B中的新值。

那么怎么知道这个共享变量的被其他线程更新了呢？这就是JMM的功劳了，也是JMM存在的必要性之一。JMM通过控制主内存与每个线程的本地内存之间的交互，来提供内存可见性保证。

7 重排序与happens- before

7.1 什么是重排序

不同的计算机体系结构具有不同的指令集，比如英特尔的 x86-64指令集。而一个计算机程序可以被解析成一系列指令。

计算机组成原理中的流水线是一种高效的指令执行方式，它将指令的执行过程划分为多个独立的阶段，并允许多个指令在不同阶段同时执行。这种并行处理的方式可以提高计算机的指令执行速度和吞吐量。

但是，流水线技术最害怕中断，恢复中断的代价是比较大的，所以我们要想尽办法不让流水线中断。指令重排就是减少中断的一种技术。

指令重排可以保证串行语义一致，但不保证多线程间的语义也一致，所以在多线程下，指令重排序可能会导致一些问题

7.2 顺序一致性模型与JMM的保证

顺序一致性模型是一个参考或者说概念，内存模型在设计的时候会参考它。

7.2.1 顺序一致性模型

顺序一致性模型有两大特性：

  • 一个线程中的所有操作必须按照程序的顺序（即Java代码的顺序）来执行。
  • 不管程序是否同步，所有线程都只能看到一个单一的操作执行顺序。即在顺序一致性模型中，每个操作必须是原子性的，且立刻对所有线程可见。

但是JMM没有这样的保证。

比如，在当前线程把写过的数据缓存在本地内存中，在没有刷新到主内存之前，这个写操作仅对当前线程可见；从其他线程的角度来观察，这个写操作根本没有被当前线程所执行。只有当前线程把本地内存中写过的数据刷新到主内存之后，这个写操作才对其他线程可见。在这种情况下，当前线程和其他线程看到的执行顺序是不一样的。

8 volatile

当一个变量被定义成volatile后，它具备两项特性：

  1. 保证此变量对所有线程的可见性
  2. 禁止指令重排序优化

9 synchronized与锁

9.1 Synchronized关键字

保证同一时刻只有一个线程可以进入同步代码块或同步方法，从而保证线程安全

实现有序性和原子性

但这个有序性和volatile实现的有序性是相对语义来看的，线程与线程间，每一个synchronized块可以看作是一个原子操作，它保证每个时刻只有一个线程执行同步代码，它可以解决工作内存和主内存同步延迟引发的无序。

  • synchronized是因为块与块之间看起来是原子操作，块与块之间有序可见
  • volatile是在底层通过内存屏障防止指令重排的，变量前后之间的指令与指令之间有序可见。

9.2.2 偏向锁

偏向锁（Biased Locking）是Java中锁优化的一种技术，旨在减少无竞争情况下的锁操作的开销。它是一种针对单线程对共享资源的独占访问的优化手段。

只要出现多个线程访问同一个临界资源，无论是否是同时访问，就会导致偏向锁的升级。

简单来说，是因为对象头里存储了锁标志位，101代表偏向锁，如果是偏向锁就去查一下当前对象头里的线程id是不是当前线程，如果是那么就继续运行，如果不是，说明出现了锁竞争要锁升级了。

上述的过程也是CAS的过程。

9.2.3 轻量级锁

多个线程在不同时段访问同一临界资源，即不存在锁竞争的情况。使用轻量级锁。

9.2.5 总结锁的升级流程

每一个线程在准备获取共享资源时： 第一步，检查MarkWord里面是不是放的自己的ThreadId ,如果是，表示当前线程是处于 “偏向锁” 。

第二步，如果MarkWord不是自己的ThreadId，锁升级，这时候，用CAS来执行切换，新的线程根据MarkWord里面现有的ThreadId，通知之前线程暂停，之前线程将Markword的内容置为空。

第三步，两个线程都把锁对象的HashCode复制到自己新建的用于存储锁的记录空间，接着开始通过CAS操作， 把锁对象的MarKword的内容修改为自己新建的记录空间的地址的方式竞争MarkWord。

第四步，第三步中成功执行CAS的获得资源，失败的则进入自旋 。

第五步，自旋的线程在自旋过程中，成功获得资源(即之前获的资源的线程执行完成并释放了共享资源)，则整个状态依然处于 轻量级锁的状态，如果自旋失败 。

第六步，进入重量级锁的状态，这个时候，自旋的线程进行阻塞，等待之前线程执行完成并唤醒自己。

9.2.6 各种锁的优缺点对比

下表来自《Java并发编程的艺术》：

锁 优点 缺点 适用场景
偏向锁 加锁和解锁不需要额外的消耗，和执行非同步方法比仅存在纳秒级的差距。 如果线程间存在锁竞争，会带来额外的锁撤销的消耗。 适用于只有一个线程访问同步块场景。
轻量级锁 竞争的线程不会阻塞，提高了程序的响应速度。 如果始终得不到锁竞争的线程使用自旋会消耗CPU。 追求响应时间。同步块执行速度非常快。
重量级锁 线程竞争不使用自旋，不会消耗CPU。 线程阻塞，响应时间缓慢。 追求吞吐量。同步块执行时间较长。
